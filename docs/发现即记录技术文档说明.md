# 发现即记录（Agent Backlog）技术文档说明

> **文档性质**：面向希望理解或复现「发现即记录」模式的同学，说明**为何需要**、**如何与 Cursor 规则配合**、**与会话交接的关系**。  
> **不是**项目日常开发必读清单；日常协作仍以仓库内 `AGENTS.md` 与 `.cursor/rules/` 为准。

---

## 1. 要解决什么问题

在 **Agentic Coding** 中，LLM 在修一个需求时，常会**顺带注意到**别的问题：潜在 bug、该重构的坏味道、缺失的测试、接口契约风险等。但当**上下文变紧**时，模型可能**不再展开这些发现**，只盯着当前任务——于是「看到过、没记下、后来忘了」，**新发现被悄悄丢弃**。

这与「会话交接」要解决的**断档**不同：交接偏**任务进度与结论**；**发现即记录**偏**观察类信息的外置**，避免依赖对话记忆。

---

## 2. 设计目标

| 目标 | 说明 |
|------|------|
| **外置化** | 发现不只靠聊天里提一句，而是落到**固定本机文件**，降低被上下文挤掉的风险。 |
| **追加优先** | **一行一条**追加，避免整文件重写导致覆盖或合并冲突。 |
| **范围清晰** | 主要记录**超出当前任务范围**、暂不处理但值得跟进的项；避免与当前 PR 改动混成一团。 |
| **与收工联动** | 在**会话交接**中增加「**观察到但未处理**」，与 `.cursor/agent-backlog.md` 对照，收工强制盘点。 |
| **不污染 Git** | 实际 backlog 数据在 **`.cursor/agent-backlog.md`**（建议 gitignore）；仓库内只保留 **`agent-backlog.template.md`** 作格式说明。 |
| **仅本机 md** | 默认**只**写入上述本机路径；不把 backlog 正文写入其他可跟踪 `.md`，除非用户明确指定。 |

---

## 3. 整体架构（与「会话交接」的配合）

```text
                    工作过程中
                         │
                         ▼
              ┌──────────────────────┐
              │ 发现超出当前任务范围   │
              │ 的事项                │
              └──────────┬───────────┘
                         │ 追加一行
                         ▼
              ┌──────────────────────┐
              │ .cursor/             │
              │   agent-backlog.md   │  ← 本机、gitignore
              └──────────┬───────────┘
                         │
         收工 / 「交接」  │  读取、概括
                         ▼
              ┌──────────────────────┐
              │ 会话交接中的        │
              │ 「观察到但未处理」   │
              └──────────────────────┘
```

**职责简述：**

1. **规则（Rule）**  
   - 在 `.cursor/rules/*.mdc` 中约定：**发现即追加**到 `agent-backlog.md`，并约定行格式。

2. **模板（Template）**  
   - `agent-backlog.template.md` 可提交仓库，只描述**格式与示例**，不含个人本机数据。

3. **本机数据文件 + gitignore**  
   - 实际条目只在 `agent-backlog.md`；**必须**加入 `.gitignore`，避免误提交敏感路径或内部备注。

4. **会话交接（姊妹流程）**  
   - 详见 **`docs/会话交接技术文档说明.md`**；交接模板中含「**观察到但未处理**」小节，与 backlog 对齐。

---

## 4. Cursor 侧机制（概念层）

- **Project Rules**  
  把「发现即记录」「追加不写满屏」「收工时对照 backlog」写进规则，**不依赖模型自觉**。

- **Commands**  
   本仓库的 **`session-handoff`**、**`agent-backlog-append`** 命令步骤中可要求：若存在 `agent-backlog.md`，先读再填交接或规范追加。

- **与 Issue / MCP**  
  若团队已用 Jira/GitLab Issue，**仍可用 backlog 作个人草稿**，再批量建工单；规则可扩展为「重要项同步到 Issue」。

---

## 5. 规则里建议写清的内容

- **何时写**：分析/修改中发现**非本任务范围**但值得记下时。  
- **写什么**：一行一条；建议 `- [日期] 简述 — 可选路径/模块`。  
- **怎么写**：**追加**为主，禁止整文件覆盖（除非用户明确要求整理归档）。  
- **收工**：交接中必须包含「**观察到但未处理**」；无则写 **无**；若已有 backlog 文件，概括或注明「与 backlog 一致」。  
- **边界**：不要借 backlog 把当前任务无限扩大；**当前任务内**该改的仍应在当轮完成。

---

## 6. 迁移到其他项目的步骤

1. 在 `.gitignore` 增加 `.cursor/agent-backlog.md`（或你的路径）。  
2. 新增可提交的 `agent-backlog.template.md`（格式说明 + 示例）。  
3. 在 Project Rules 中增加「发现即记录」小节（追加策略、仅本机 md）。  
4. 在会话交接规则/模板中增加「观察到但未处理」小节。  
5. 可选：在 `session-handoff` 类命令中增加「先读 backlog」步骤。

---

## 7. 使用与注意事项

- **隐私**：backlog 可能含内部路径、模块名；gitignore + 勿把整份贴到公开渠道。  
- **准确性**：条目是**观察**，不是结论；**是否真改**仍以代码审查与 CI 为准。  
- **与 Memory**：Cursor Memory 偏长期偏好；backlog 偏**当前项目、任务周边的待跟进观察**，二者可并存。  
- **清理**：可定期把已处理行标 `[done]` 或移到「已处理」区，避免文件无限增长（规则可约定由用户或 Agent 在整理会话中处理）。

---

## 8. 局限与改进方向

| 局限 | 可能改进 |
|------|----------|
| 仍依赖 Agent 执行追加 | 规则写死 + 收工交接二次盘点；重要项人工复制到 Issue。 |
| 文件变长 | 分文件（如按月份）或定期归档；规则中约定上限与归档方式。 |
| 与「当前任务」界限模糊 | 在规则里举例：「本需求内的 bug 当场修；别模块的坏味道进 backlog」。 |
