# 任务计划（Task Plan）技术文档说明

> **文档性质**：说明如何用**上下文外**的结构化计划缓解 **「过早宣告完成」**（尤其经历多次 compaction 后）。  
> **不是**项目日常开发必读清单；日常协作仍以 `AGENTS.md` 与 `.cursor/rules/` 为准。

---

## 1. 要解决什么问题

Agent 在长对话、多次 **compaction** 后，可能**丢失早期约定的阶段划分**，仅凭近期消息就断言「任务全部完成」，而分析、测试、联调等阶段尚未做。

**任务计划**把**整体目标、分阶段验收、勾选状态**固定在**本机文件**，与对话记忆解耦；收工或汇报前**先读文件再说话**。

---

## 2. 设计目标

| 目标 | 说明 |
|------|------|
| **计划外置** | 真值在 `.cursor/task-plan.md`，不依赖被压缩的上下文。 |
| **结构化** | 阶段列表 + `[ ]`/`[x]`，避免口头「差不多了」。 |
| **对照机制** | 用户或规则触发「对照计划」；交接中必须汇报计划进度。 |
| **完成宣告约束** | 仍有未勾选阶段时，**不得**宣称整体任务完成。 |
| **仅本机** | 数据文件 gitignore；仓库内只保留 `task-plan.template.md`。 |

---

## 3. 与「发现即记录」「会话交接」的关系

| 机制 | 侧重 |
|------|------|
| **task-plan** | **主线阶段**是否做完、能否宣称整体完成。 |
| **agent-backlog** | **范围外顺带发现**，不替代阶段勾选。 |
| **会话交接** | 收工时汇总，含 **任务计划进度** + **观察到但未处理**。 |

## 5. 使用案例

### 1. 任务名称

用一句话能区分不同需求即可，例如：订单列表支持下拉刷新与空态。  
避免过泛（如「优化订单」），否则阶段不好切。

### 2. 目标与验收（整体）

写清两件事：

- **目标**：做完后用户/业务上能多出什么能力或修掉什么问题。  
- **验收（整体）**：怎样算「这个大需求可以收口」——可测、可演示，例如：`dart analyze` / `flutter analyze` 无新增 error；列表在空/错/正常三种情况可验证。  

不必写实现细节，写结果与可验证标准即可。

### 3. 阶段列表（完成打勾）

这是核心，建议：

- 按依赖顺序拆：先能跑通主路径，再补边界、再补测试/文档。  
- 每行一条阶段，格式保持：**阶段 N：做什么 — 验收：怎样算这一阶段完成**。  
- 阶段粒度：小到 Agent 一两轮能做完一档，避免一行里塞太多事。

示例（仅示意）：

- [ ] 阶段 1：列表页接入接口与基础列表展示 — 验收：有数据时能列出，无崩溃  
- [ ] 阶段 2：下拉刷新与加载失败重试 — 验收：手动可触发刷新与重试  
- [ ] 阶段 3：空态与错误态 UI — 验收：与产品稿一致或团队认可  
- [ ] 阶段 4：analyze + 自测路径写进交接 — 验收：analyze 通过，交接可复述测试步骤  

完成某一阶段后，把该行 `[ ]` 改成 `[x]`，并改「当前进度」。

### 4. 当前进度（一句话）

给 compaction 后快速用的，例如：阶段 2 已完成，在做阶段 3。  
每次勾选阶段后顺手更新。

### 5. 文件怎么落地

复制 `.cursor/task-plan.template.md` 为 `.cursor/task-plan.md`（本机，不提交）。  
删掉模板里占位的 `…`，换成你的真实阶段与验收。  
或直接在对话里说：「按 task-plan.template 帮我生成 `.cursor/task-plan.md`，任务是：……」让 Agent 起草，你再改阶段拆分。

### 6. 写好计划书的要点

| 要点 | 说明 |
|------|------|
| 验收可执行 | 每阶段「验收」最好是能操作或能跑命令验证的，不要只写「完成开发」。 |
| 阶段数适中 | 一般 3～7 段；太碎难维护，太粗容易又「假完成」。 |
| 和 backlog 分工 | 计划管主线阶段；顺带发现、暂不做的记在 `agent-backlog`，不要全塞进阶段里。 |
